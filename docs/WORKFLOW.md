# 项目启动详细流程

本文档详细描述了 project-init 的完整工作流程。

## 流程概览

```
Phase 1: Requirements Clarification
    ↓
Phase 2: PRD Generation
    ↓
Phase 3: MVP Definition
    ↓
Phase 4: Technical Architecture
    ↓
Phase 5: Development Roadmap
```

---

## Phase 1: Requirements Clarification

### 目标
通过渐进式提问理解项目的业务目标、用户需求和约束条件。

### 流程

#### Step 1: 捕获初始上下文
如果用户提供了激活参数，提取：
- 项目类型（Web/Mobile/API/Desktop/Other）
- 项目名称
- 一句话描述

如果没有提供参数，询问：
```
请告诉我：
1. 项目类型：[Web应用/移动应用/API服务/桌面应用/其他]
2. 项目名称：[你的项目名]
3. 一句话描述：[用一句话描述这个项目要解决什么问题]
```

#### Step 2: 深度提问

按顺序提问，根据回答深入探讨。使用这 15 个核心问题：

**业务目标（必问）**
1. **核心问题**：这个项目要解决什么具体痛点？成功标准是什么？
2. **目标用户**：主要用户群体是谁？他们的技术熟练度如何？
3. **价值主张**：与现有解决方案相比，独特价值是什么？
4. **商业模式**：如何盈利或创造价值？（订阅/广告/交易抽成/效率提升）

**功能需求（渐进式）**
5. **核心场景**：描述一个典型的用户使用流程（User Story）
6. **功能边界**：哪些功能是"必须有"，哪些是"最好有"？
7. **内容策略**：内容由谁产生？UGC/PGC/AI生成/抓取？
8. **集成需求**：需要对接第三方服务吗？（支付/地图/社交登录/AI API）

**技术约束（关键）**
9. **性能要求**：预期的用户规模？并发量？响应时间要求？
10. **平台约束**：必须支持哪些平台？（iOS/Android/桌面/特定浏览器）
11. **合规要求**：是否有 GDPR/CCPA/等保/行业特定合规要求？
12. **技术偏好**：团队技术栈偏好？是否有必须使用的技术？

**项目管理（实际）**
13. **时间约束**：期望的上线时间？是否有硬性截止日期？
14. **资源限制**：团队规模？预算范围？是否需要开源方案？
15. **维护计划**：上线后的迭代频率？运维能力如何？

#### Step 3: 总结并确认

收集完所有信息后，总结关键点：
```
## 需求澄清总结

**项目**：[名称] - [类型]
**核心问题**：[一句话]
**目标用户**：[描述]
**关键约束**：[技术/时间/资源]

是否准确？需要调整什么？
```

等待用户确认后再进入 Phase 2。

## Phase 2: PRD Generation

### 目标
基于澄清的需求创建标准化的产品需求文档。

### 输出
生成 `docs/PRD.md`，包含以下结构：

```markdown
# 产品需求文档 (PRD)

## 1. 项目概述
- 项目名称：
- 版本：0.1.0
- 日期：YYYY-MM-DD
- 状态：草案

## 2. 背景与目标
### 2.1 问题陈述
[从 Phase 1 提炼]

### 2.2 成功指标 (KPI)
- [可量化的指标]

## 3. 用户画像
| 角色 | 年龄/职业 | 目标 | 痛点 | 技术熟练度 |
|------|----------|------|------|-----------|
| 主要用户 | | | | |
| 次要用户 | | | | |

## 4. 功能需求
### 4.1 核心功能 (Must Have)
- [ ] 功能1：描述 + 验收标准
- [ ] 功能2：描述 + 验收标准

1. 使用上述结构创建文件
2. 基于 Phase 1 的回答填充内容
3. 呈现给用户审阅
4. 根据反馈迭代
5. 确认"PRD 已定稿"后再继续

## Phase 3: MVP Definition

### 目标
通过筛选 Must Have vs Nice to Have 功能严格定义 MVP 范围。

### MVP 标准
对每个潜在功能，问三个问题：
1. **没有这个，用户能完成核心任务吗？**（如果不能 → Must Have）
2. **去掉这个，产品还完整吗？**（如果否 → Must Have）
3. **实现这个需要超过 30% 开发时间吗？**（如果是 → 考虑移到 Phase 2）

### 输出
生成 `docs/MVP.md`：

```markdown
# MVP 定义文档

## 版本范围
- 版本号：v0.1.0
- 目标日期：YYYY-MM-DD
- 核心理念：用最小成本验证核心假设

## 包含功能 (In Scope)
| 功能模块 | 具体功能 | 业务价值 | 预估工时 |
|---------|---------|---------|---------|
| 用户系统 | 注册/登录/找回密码 | 身份识别 | 8h |
| [模块名] | [功能描述] | [价值] | [工时] |

## 明确排除 (Out of Scope)
| 功能 | 排除原因 | 计划版本 |
|------|---------|---------|
| [功能名] | [原因] | [版本] |

## 技术债务预警
- 为赶进度暂时采用的方案：
- 需要在 v0.2.0 前重构的部分：

## 验收标准
- [ ] 标准1
- [ ] 标准2
```

### 流程
1. 根据 MVP 标准审查 PRD 功能
2. 将每个功能分类：In Scope / Out of Scope
3. 记录排除的理由
4. 定义验收标准
5. 获得用户确认

## Phase 4: Technical Architecture

### 目标
推荐技术栈并生成项目结构。

### 技术栈决策矩阵

基于项目需求，从以下选项中推荐：

| 维度 | 选项 | 选择标准 |
|-----------|---------|-------------------|
| 前端 | React/Vue/Svelte/Next/Nuxt | SEO 需求、团队熟悉度 |
| 样式 | Tailwind/Styled Components/CSS Modules | 设计系统复杂度 |
| 后端 | Node/Go/Python/Rust | 性能需求、生态系统 |
| 数据库 | PostgreSQL/MongoDB/Supabase/Firebase | 数据关系、实时性 |
| 部署 | Vercel/Netlify/AWS/Docker | 流量预期、预算 |

### 输出

#### 1. 技术栈推荐
呈现 2-3 个方案对比：

```markdown
## 技术栈方案对比

### 方案 A: [名称]
- 前端：[技术] - [理由]
- 后端：[技术] - [理由]
- 数据库：[技术] - [理由]
- 部署：[技术] - [理由]
- **适合场景**：[描述]

### 方案 B: [名称]
...

**推荐**：方案 [A/B/C] - [理由]
```

#### 2. 项目结构
基于选定的技术栈生成目录结构。

全栈 Web 应用示例：
```
my-project/
├── apps/
│   ├── web/                    # Next.js 前端
│   │   ├── src/
│   │   │   ├── app/           # App Router
│   │   │   ├── components/    # React 组件
│   │   │   ├── hooks/        # 自定义 Hooks
│   │   │   ├── lib/          # 工具函数
│   │   │   └── types/        # TypeScript 类型
│   │   └── public/
│   └── api/                    # 后端服务
│       ├── src/
│       │   ├── routes/        # API 路由
│       │   ├── controllers/   # 业务逻辑
│       │   ├── models/        # 数据模型
│       │   └── services/      # 外部服务
├── packages/
│   └── shared-types/           # 共享类型
├── docs/
│   ├── PRD.md
│   ├── MVP.md
│   └── API.md
└── README.md
```

#### 3. 编辑器配置文件

为用户的编辑器生成配置文件：

**Cursor (.cursorrules)**：
```text
# 项目规范
- 使用 TypeScript 严格模式
- 组件使用函数式组件 + Hooks
- 所有 API 调用封装到 services/ 目录
- 错误处理必须使用 try-catch
- 提交前必须运行 lint 和 type-check

# 代码风格
- 优先使用 named exports
- 组件文件使用 PascalCase
- 工具函数使用 camelCase
- 常量使用 UPPER_SNAKE_CASE
```

**Windsurf (.windsurfrules)**：
```text
[架构]
- 前端: [技术栈]
- 后端: [技术栈]
- 样式: [方案]

[约束]
- [具体约束1]
- [具体约束2]
```

**Trae (.trae/rules.yaml)**：
```yaml
project_type: [类型]
tech_stack:
  frontend: [技术]
  backend: [技术]
  database: [数据库]
conventions:
  component_style: [风格]
  api_style: [风格]
  testing: [要求]
```

### 流程
1. 分析 PRD/MVP 的需求
2. 推荐技术栈并说明理由
3. 获得用户选择
4. 生成项目结构
5. 创建编辑器配置文件
6. 确认后再继续

## Phase 5: Development Roadmap

### 目标
创建包含里程碑和时间估算的开发路线图。

### 时间估算方法
使用**三点估算法**（乐观/最可能/悲观）：

公式：`估算 = (乐观 + 4×最可能 + 悲观) / 6`

### 输出
生成 `docs/ROADMAP.md`：

```markdown
# 开发路线图

## 里程碑 1: 基础架构 (Week 1)
- [ ] 项目脚手架搭建 (4h/6h/10h)
- [ ] 数据库设计 + 迁移 (6h/8h/12h)
- [ ] CI/CD 流程配置 (4h/6h/8h)
- **缓冲时间**：20%
- **交付物**：可运行的 Hello World + 数据库连接

## 里程碑 2: 核心功能 (Week 2-3)
- [ ] 用户认证系统 (8h/12h/16h)
- [ ] 主业务功能 A (16h/24h/32h)
- [ ] 主业务功能 B (12h/16h/24h)
- **缓冲时间**：25%
- **交付物**：内部测试版本 (Alpha)

## 里程碑 3: MVP 完善 (Week 4)
- [ ] 错误处理与日志 (6h/8h/12h)
- [ ] 基础管理后台 (8h/12h/16h)
- [ ] 部署与监控 (4h/6h/10h)
- **缓冲时间**：15%
- **交付物**：公测版本 (Beta)

## 关键路径
[描述依赖关系和关键路径]

## 风险缓解
| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| [风险1] | 高/中/低 | 高/中/低 | [策略] |
| [风险2] | 高/中/低 | 高/中/低 | [策略] |

## 验收清单
- [ ] PRD.md 已生成并通过评审
- [ ] MVP.md 已明确功能边界
- [ ] 项目目录结构已创建
- [ ] 编辑器配置文件已就位
- [ ] ROADMAP.md 已制定并达成共识
- [ ] 首次 commit 已完成（README + 基础配置）
```

### 流程
1. 将 MVP 功能分解为任务
2. 对每个任务应用三点估算
3. 分组为里程碑（通常 3-4 个）
4. 添加缓冲时间（每个里程碑 15-25%）
5. 识别关键路径和依赖关系
6. 列出前 3 个风险及缓解策略
7. 呈现给用户并迭代

---

## 完成标准

项目启动完成的标志：
- ✅ PRD.md 已生成并审阅
- ✅ MVP.md 定义了清晰的边界
- ✅ 项目结构已创建
- ✅ 编辑器配置文件已就位
- ✅ ROADMAP.md 已达成共识
- ✅ 初始 commit 已完成

## 注意事项

- 始终在进入下一阶段前确认当前阶段
- 根据项目类型（Web/Mobile/API/Desktop）调整问题
- 保持灵活 - 如果用户说"跳到架构"，尊重他们的选择
- 专注于实际成果而非僵化的流程遵守
- 在生成的文件中记录所有决策
