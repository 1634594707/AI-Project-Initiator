# 最佳实践

基于实际使用经验总结的最佳实践和常见陷阱。

## 需求澄清阶段（Phase 1）

### ✅ 推荐做法

**1. 准备充分的上下文**
- 提前整理用户反馈、竞品分析、市场调研
- 准备好相关的原型图、设计稿、技术文档
- 明确团队的技术栈偏好和约束

**2. 诚实回答"不确定"**
- 不确定时直接说"不确定"，AI 会提供选项
- 避免为了快速推进而给出模糊答案
- 标记需要后续调研的问题

**3. 关注约束而非解决方案**
- 描述问题和约束，而非具体实现
- 说"需要支持 10 万并发"而非"需要用 Redis"
- 让 AI 基于约束推荐方案

### ❌ 常见陷阱

**1. 跳过需求澄清直接要架构**
- 问题：缺少上下文导致架构不匹配需求
- 解决：至少回答核心的 5-8 个问题

**2. 过度设计 MVP**
- 问题：把所有想要的功能都塞进 MVP
- 解决：严格应用"没有这个用户能完成核心任务吗"标准

**3. 忽略非功能需求**
- 问题：只关注功能，忽略性能、安全、合规
- 解决：明确性能指标、安全要求、合规标准

## PRD 编写阶段（Phase 2）

### ✅ 推荐做法

**1. 使用 @doc-coauthor 协作**
- 结构化的写作流程确保文档质量
- 通过"读者测试"发现盲点
- 迭代式完善而非一次性完成

**2. 明确成功指标**
- 每个功能都有可量化的验收标准
- 定义 KPI 和测量方法
- 设定合理的目标值

**3. 详细的用户画像**
- 不只是"年轻人"，要具体到职业、场景、痛点
- 区分主要用户和次要用户
- 包含技术熟练度评估

### ❌ 常见陷阱

**1. PRD 写成技术方案**
- 问题：过早陷入技术细节
- 解决：PRD 聚焦"做什么"，架构文档聚焦"怎么做"

**2. 缺少排除项**
- 问题：没有明确说"不做什么"
- 解决：明确列出 Won't Have 清单

**3. 用户流程过于抽象**
- 问题：流程图看不出实际操作
- 解决：用具体的用户故事描述流程

---

## MVP 定义阶段（Phase 3）

### ✅ 推荐做法

**1. 使用 @pm-kit 做 RICE 排序**
- 量化每个功能的价值和成本
- 识别快速胜利（Quick Wins）
- 平衡短期和长期价值

**2. 严格的三问标准**
- 没有这个，用户能完成核心任务吗？
- 去掉这个，产品还完整吗？
- 实现这个需要超过 30% 开发时间吗？

**3. 记录排除理由**
- 每个排除的功能都说明原因
- 标注计划在哪个版本实现
- 避免未来重复讨论

### ❌ 常见陷阱

**1. MVP 等于"简陋版"**
- 问题：为了快而牺牲核心体验
- 解决：MVP 应该是"最小但完整"的产品

**2. 技术债务失控**
- 问题：为赶进度采用临时方案但没记录
- 解决：明确标注技术债务和重构计划

**3. 没有验收标准**
- 问题：MVP 完成标准模糊
- 解决：列出清晰的验收清单

---

## 技术架构阶段（Phase 4）

### ✅ 推荐做法

**1. 对比多个方案**
- 至少提供 2-3 个技术栈方案
- 说明每个方案的优劣和适用场景
- 基于约束做决策而非个人偏好

**2. 生成完整的项目结构**
- 不只是顶层目录，要到文件级别
- 包含配置文件、测试目录、文档目录
- 符合技术栈的最佳实践

**3. 配置编辑器规则**
- 生成 .cursorrules / .windsurfrules / .trae/rules.yaml
- 包含代码风格、命名约定、架构约束
- 让 AI 编辑器理解项目规范

### ❌ 常见陷阱

**1. 选择最新但不成熟的技术**
- 问题：追求新技术导致生态不完善
- 解决：优先选择成熟稳定的技术栈

**2. 过度工程化**
- 问题：为未来可能的需求设计复杂架构
- 解决：YAGNI 原则，只设计当前需要的

**3. 忽略团队能力**
- 问题：选择团队不熟悉的技术
- 解决：考虑学习成本和招聘难度

## 开发路线图阶段（Phase 5）

### ✅ 推荐做法

**1. 使用三点估算法**
- 给出乐观/最可能/悲观三个时间
- 计算加权平均：(乐观 + 4×最可能 + 悲观) / 6
- 比拍脑袋估算更准确

**2. 添加缓冲时间**
- 每个里程碑预留 15-25% 缓冲
- 考虑会议、评审、返工时间
- 为意外情况留余地

**3. 识别关键路径**
- 标注任务依赖关系
- 找出影响整体进度的关键任务
- 优先保障关键路径的资源

**4. 风险管理**
- 列出 top 3 风险
- 评估概率和影响
- 制定具体的缓解策略

### ❌ 常见陷阱

**1. 过于乐观的估算**
- 问题：只考虑编码时间，忽略测试、调试、返工
- 解决：估算时间 × 1.5 作为实际预期

**2. 忽略依赖关系**
- 问题：并行任务实际有依赖导致阻塞
- 解决：明确标注前置依赖

**3. 没有里程碑验收标准**
- 问题：里程碑完成标准模糊
- 解决：每个里程碑定义清晰的交付物

## 跨阶段最佳实践

### 文档版本控制

**推荐**：
- 所有文档纳入 Git 版本控制
- 重大变更记录在文档的"变更历史"部分
- 使用 PR 流程评审文档变更

### 团队协作

**推荐**：
- PRD 完成后组织评审会议
- 技术架构需要工程团队确认可行性
- MVP 范围需要产品、设计、工程三方对齐

### 持续更新

**推荐**：
- 开发过程中发现的问题及时更新文档
- 每个迭代结束后回顾路线图
- 保持文档与实际开发同步

## 扩展 Skills 使用建议

### 何时使用 @pm-kit

**适合**：
- 功能列表超过 10 个需要排序
- 多个利益相关方有不同优先级诉求
- 需要量化决策依据

**不适合**：
- 功能很少（< 5 个）且优先级明显
- 时间紧迫需要快速决策

### 何时使用 @doc-coauthor

**适合**：
- 文档需要多人评审
- 文档会被广泛传播（团队、投资人、合作伙伴）
- 需要高质量的文档输出

**不适合**：
- 内部快速沟通的草稿
- 一次性的临时文档

### 何时使用 @web-test

**适合**：
- Web 应用有复杂的用户交互
- 需要回归测试保证质量
- 团队有自动化测试文化

**不适合**：
- 纯后端 API 服务（用 API 测试工具）
- 静态网站（手动测试即可）

### 何时使用 @ux-kit

**适合**：
- 需要统一的设计系统
- 多个页面/组件需要一致的视觉风格
- 团队缺少专业设计师

**不适合**：
- 已有成熟的设计系统
- 项目对 UI 要求不高

## 效率提升技巧

### 1. 批量回答问题
不用等 AI 一个个问，可以一次性回答多个：
```
1. 目标用户是企业管理者
2. 核心痛点是数据分散难以统计
3. 需要支持 Web 和移动端
4. 预算有限，优先开源方案
```

### 2. 引用已有文档
如果你已经有部分文档：
```
我已经有一份初步的需求文档，内容如下：
[粘贴内容]

请基于这个生成标准的 PRD
```

### 3. 使用模板快速填充
直接复制模板到你的项目，填写关键部分，让 AI 帮你完善：
```
我已经填写了 PRD 模板的 1-4 部分，请帮我完成 5-8 部分
```

## 进阶用法

### 自定义预设

复制现有预设并修改：
```yaml
name: my-custom
description: 我的自定义预设
skills:
  - project-init
  - pm-kit
  - web-test  # 只用我需要的
```

### 集成到 CI/CD

将生成的文档作为项目的"单一事实来源"：
- 在 PR 中检查代码是否符合架构约束
- 在发布前检查是否完成 MVP 验收清单
- 定期审查路线图与实际进度的偏差

### 多项目复用

建立你的项目模板库：
- 保存常用的技术栈组合
- 复用编辑器配置文件
- 积累领域特定的问题清单
